# What Launch 프로젝트 개발 규칙

## 프로젝트 개요
- **목적**: "오늘 점심 뭐먹지?" 고민을 가볍게 해소하는 뽑기 게임 형태의 웹 서비스
- **핵심 원칙**: 최소한의 비용으로 운영하며 개발자 한 명이 관리할 수 있는 심플한 구조

## 제약사항 (반드시 준수)

### 비용 제약
- **월 운영비용: $5 이하** (무료 티어 최대 활용)
- 추가 라이브러리는 무료 티어 범위 내에서만 사용
- API 호출은 무료 티어 내에서 사용 (카카오맵: 일 300,000건)
- 호스팅은 GitHub Pages 등 완전 무료 서비스 사용

### 시간 제약
- **개발 기간: 1주 이내**
- MVP 기능 우선 구현, 선택 기능은 후순위
- 복잡한 기능보다는 단순하고 빠른 구현 우선

### 인력 제약
- **유지보수 인력: 1명**
- 코드는 간결하고 이해하기 쉬워야 함
- 과도한 추상화나 복잡한 아키텍처 지양

### 트래픽 제약
- 초기 예상 트래픽: 월 100명 방문자 이하
- 성능 최적화는 필요하지만 과도한 최적화는 불필요

## 기술 스택 규칙

### 필수 사용 기술
- **프론트엔드**: React + Vite (Next.js 사용 금지 - 과함)
- **언어**: TypeScript
- **스타일링**: Tailwind CSS
- **상태 관리**: React Context API + useReducer (추가 라이브러리 지양)
- **HTTP 클라이언트**: Fetch API (네이티브, 추가 라이브러리 지양)
- **폼 관리**: Native HTML Form + React State (React Hook Form 등 추가 라이브러리 지양)

### 지도 API
- **카카오맵 API** 우선 사용
- JavaScript SDK: 지도 렌더링
- REST API: 주소 검색, 키워드로 장소 검색
- 무료 티어 범위 내에서만 사용

### 금지 사항
- ❌ Next.js, Remix 등 SSR/SSG 프레임워크 (불필요)
- ❌ Redux, Zustand 등 상태 관리 라이브러리 (Context API로 충분)
- ❌ React Hook Form, Formik 등 폼 라이브러리 (단순 입력 필드 1개)
- ❌ Axios 등 HTTP 클라이언트 라이브러리 (Fetch API로 충분)
- ❌ 과도한 UI 라이브러리 (Tailwind CSS로 충분)

## 코드 작성 규칙

### 일반 원칙
1. **단순함 우선**: 복잡한 패턴보다는 단순하고 이해하기 쉬운 코드
2. **타입 안정성**: TypeScript를 적극 활용, `any` 타입 최소화
3. **컴포넌트 분리**: 재사용 가능한 컴포넌트는 `src/components/`에 분리
4. **유틸리티 함수**: 공통 로직은 `src/lib/`에 분리
5. **타입 정의**: 모든 타입은 `src/types/index.ts`에 정의

### 파일 구조 준수
```
src/
├── components/     # 재사용 가능한 컴포넌트
├── lib/            # 유틸리티 및 헬퍼 함수
├── context/        # React Context
└── types/          # TypeScript 타입 정의
```

### 네이밍 규칙
- 컴포넌트: PascalCase (예: `LocationInput.tsx`)
- 함수/변수: camelCase (예: `getRestaurantList`)
- 타입/인터페이스: PascalCase (예: `Restaurant`, `AppState`)
- 상수: UPPER_SNAKE_CASE (예: `MAX_SEARCH_RADIUS`)

### 스타일링 규칙
- **Tailwind CSS만 사용** (커스텀 CSS 최소화)
- 반응형 디자인 필수 (`sm:`, `md:`, `lg:` 등 활용)
- 모바일 우선 설계 (Mobile First)

## 기능 구현 규칙

### 1. 위치 입력 기능 (FR-001 ~ FR-003)
- 주소 또는 장소명 입력 가능
- 지도 API를 통해 좌표로 변환
- 자동완성 기능은 선택사항 (구현 시간 고려)

### 2. 음식 장르 선택 기능 (FR-004 ~ FR-006)
- 지원 장르: 한식, 일식, 중식, 양식, 기타
- "랜덤" 옵션 제공 (장르까지 랜덤 선택)
- 단일 선택 방식 (라디오 버튼 또는 버튼 그룹)

### 3. 식당 검색 및 추천 기능 (FR-007 ~ FR-010)
- 입력된 위치 기준 주변 식당 검색
- 선택된 음식 장르에 맞는 식당만 필터링
- 검색된 식당 중 랜덤으로 하나 선택
- 검색 반경: 기본값 1km, 최대 5km

### 4. 지도 시각화 기능 (FR-011 ~ FR-015)
- 입력한 주소 위치 마커 표시
- 추천된 식당 위치 마커 표시
- 두 마커는 서로 다른 색상/아이콘으로 구분
- 카카오맵 형태로 렌더링
- 반응형 디자인 (모바일/데스크톱 모두 지원)

### 5. 다시 선택하기 기능 (FR-016 ~ FR-018)
- "다시 선택하기" 버튼으로 새로운 식당 추천
- 동일한 위치와 장르 조건으로 재검색
- 이전 추천 식당 제외는 선택사항 (구현 시간 고려)

### 6. 식당 정보 표시 (FR-019)
- 식당명, 주소, 거리 필수 표시
- 전화번호, 평점은 있는 경우만 표시

## 성능 요구사항 (NFR-001 ~ NFR-003)

### 반드시 준수
- 지도 렌더링 시 끊김 없이 부드럽게 표시
- 식당 검색 응답 시간: **3초 이내**
- 페이지 로딩 시간: **2초 이내**

### 최적화 전략
- 지도 지연 로딩: 사용자가 검색할 때만 지도 로드
- API 호출 최적화: 디바운싱 및 캐싱 활용
- 코드 스플리팅: Vite의 자동 코드 스플리팅 활용
- 번들 최적화: 불필요한 라이브러리 제거

## UI/UX 요구사항 (UI-001 ~ UI-008)

### 화면 구성
1. 위치 입력 필드
2. 음식 장르 선택 영역
3. "랜덤" 선택 옵션
4. "추천받기" 버튼
5. 지도 영역
6. 추천된 식당 정보 표시 영역
7. "다시 선택하기" 버튼

### 인터랙션
- 위치 입력 후 장르 선택 가능
- 장르 선택 후 "추천받기" 버튼 활성화
- 추천 결과 표시 시 지도 자동으로 적절한 줌 레벨로 조정
- "다시 선택하기" 버튼 클릭 시 즉시 새로운 추천 제공

### 디자인
- **심플하고 깔끔한 디자인** (과도한 장식 지양)
- 주요 액션 버튼은 명확하게 구분
- 반응형 디자인 필수

## 에러 처리 규칙

### 필수 처리
- API 호출 실패 시 사용자에게 명확한 에러 메시지 표시
- 네트워크 오류 처리
- 잘못된 위치 입력 처리
- 검색 결과 없음 처리

### 에러 메시지
- 사용자 친화적인 한국어 메시지
- 기술적인 에러는 콘솔에만 출력
